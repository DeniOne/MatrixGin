────────────────────────────────────────
MODULE PRODUCTION PROTOCOL (MPP)
────────────────────────────────────────

Любой новый модуль MatrixGin производится
строго по Module Production Protocol (MPP).

MPP — это НЕ рекомендация,
а обязательный порядок мышления и действий.

Модуль НЕ считается существующим,
если он не прошёл все шаги MPP.


ШАГ 0. MODULE IDEA (ДО КОДА)

Фиксируется кратко:
— название модуля
— какую человеческую ценность он усиливает
— что модуль ПРИНЦИПИАЛЬНО НЕ делает


ШАГ 1. STRATEGIC CHECK (ИНВАРИАНТЫ)

Модуль проверяется на соответствие:
— этике MatrixGin
— гуманистическим принципам
— роли AI (advisory only)
— отсутствию скрытой автоматизации и власти

Если есть сомнения — модуль DEFERRED.


ШАГ 2. ARCHITECTURE SLOT

Фиксируется архитектурное место модуля:
— слой системы
— зависимости
— кто может использовать модуль
— какие зависимости ЗАПРЕЩЕНЫ

Без Architecture Slot код писать запрещено.


ШАГ 3. CONTRACT DRAFT

До реализации фиксируются границы:
— API (если есть)
— Registry / сущности (если есть)
— ACL / доступы (если есть)

Контракт важнее реализации.


ШАГ 4. FUNCTIONAL SCENARIOS

Фиксируются:
— минимум 2 допустимых сценария
— минимум 2 запрещённых сценария

Это защита от логических и этических ошибок.


ШАГ 5. IMPLEMENTATION

Только после шагов 0–4:
— разрешена реализация кода
— строго в рамках утверждённых документов
— без расширения scope


ШАГ 6. VALIDATION & STATUS

По итогам модуль получает статус:
— READY — можно развивать и отдавать
— LIMITED — только внутреннее развитие
— BLOCKED — развитие запрещено до редизайна

Статус модуля фиксируется явно.


Любой пропуск шага MPP
делает модуль НЕКОРРЕКТНЫМ,
даже если код работает.


Каждый шаг MPP обязан быть
отражён в артефактах модуля:

— Шаги 0–1 → MODULE-SPEC.md
— Шаги 2–3 → DEVELOPMENT-CHECKLIST.md
— Шаг 4 → DEVELOPMENT-CHECKLIST.md (Scenarios)
— Шаг 5 → IMPLEMENTATION-CHECKLIST.md
— Шаг 6 → явный статус модуля (READY / LIMITED / BLOCKED)

Если шаг не отражён в документах —
считается, что шаг НЕ выполнен.


Каждый модуль MatrixGin ОБЯЗАН
иметь явно зафиксированный статус:

— READY
— LIMITED
— BLOCKED

Отсутствие статуса приравнивается к BLOCKED.

────────────────────────────────────────
МОДУЛЬНАЯ СТРУКТУРА (ОБЯЗАТЕЛЬНО)
────────────────────────────────────────

— Все модули хранятся в каталоге /01-modules
— Каждый модуль:
  • имеет УНИКАЛЬНЫЙ порядковый номер
  • номер = логический приоритет разработки
  • имя отражает суть модуля

— В КАЖДОМ модуле ОБЯЗАТЕЛЬНО:

  1. MODULE-SPEC.md
     • философия и смысл
     • этическая роль
     • эмоциональное влияние
     • связи
     • границы и запреты

     MODULE-SPEC.md фиксирует ЗАЧЕМ существует модуль
     и какие действия в нём ЗАПРЕЩЕНЫ навсегда.

     Делай единым текстом в формате .md,
     чтобы можно было скопировать и вставить.

  2. DEVELOPMENT-CHECKLIST.md
     • фазовая логика
     • архитектурные и security checkpoints
     • этические guardrails
     • ограничения для Claude Opus и AI

     DEVELOPMENT-CHECKLIST.md определяет,
     ИМЕЕМ ЛИ МЫ ПРАВО реализовывать и расширять модуль.

     Делай единым текстом в формате .md,
     чтобы можно было скопировать и вставить.

  3. IMPLEMENTATION-CHECKLIST.md
     • конкретные инженерные задачи
     • реальные TODO по коду
     • MUST / SHOULD / OPTIONAL / DEFERRED
     • критерии закрытия модуля

     IMPLEMENTATION-CHECKLIST.md отвечает на вопрос:
     «ЧТО КОНКРЕТНО ДЕЛАЕМ РУКАМИ».

     IMPLEMENTATION-CHECKLIST.md:
     — НЕ может противоречить MODULE-SPEC.md
     — НЕ может нарушать DEVELOPMENT-CHECKLIST.md
     — используется для программирования и закрытия модуля

     Модуль НЕ считается реализованным,
     пока не закрыты все MUST DO пункты
     IMPLEMENTATION-CHECKLIST.md.

     Делай единым текстом в формате .md,
     чтобы можно было скопировать и вставить.

Никакого кода без утверждённого смысла.
Никакой эффективности ценой человека.
