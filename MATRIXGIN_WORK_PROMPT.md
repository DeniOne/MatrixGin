────────────────────────────────────────
БАЗОВОЕ ОПРЕДЕЛЕНИЕ (КАНОН)
────────────────────────────────────────

MatrixGin (MG) — это ЭТИЧЕСКАЯ СИСТЕМА.

MG — это система, которая не просто повышает эффективность,
а формирует этичного человека
в общепринятом гуманистическом смысле.

Этика в MatrixGin:
— не является идеологией
— не навязывает мораль
— не оценивает людей

Этика в MG — это:
осознанность, ответственность и понимание последствий своих действий
по отношению к себе, другим людям и системе в целом.

────────────────────────────────────────
КОНТЕКСТ
────────────────────────────────────────

MatrixGin — это не просто ERP или корпоративная система.
Это эмоциональная операционная система компании,
в которой ключевой целью является не контроль,
а создание среды, где человеку ХОЧЕТСЯ работать,
быть вовлечённым, расти и создавать ценность.

────────────────────────────────────────
ОБЯЗАТЕЛЬНЫЕ ПРАВИЛА РАБОТЫ
────────────────────────────────────────

1. MatrixGin = ЭТИЧЕСКАЯ СИСТЕМА
2. НИ ОДНОЙ строки кода без:
   — MODULE-SPEC.md
   — DEVELOPMENT-CHECKLIST.md
   — указания PHASE
3. Сначала смысл и архитектура → потом код
4. Не расширять scope
5. Не нарушать фазовую логику
6. AI — advisory only
7. Emotional Guardrail обязателен
8. Dialogue Capture & Consent — непроходимый этический контур
9. Если модуль не готов по зависимостям — он DEFERRED
10. Каждый модуль MatrixGin доводится до РЕАЛЬНОГО ИСПОЛНЕНИЯ
    ПЕРЕД переходом к следующему модулю.

    Реальное исполнение модуля означает:
    — утверждённый MODULE-SPEC.md
    — утверждённый DEVELOPMENT-CHECKLIST.md
    — рабочий код, соответствующий этим документам
    — отсутствие открытых MUST-DO задач по модулю
    — OPTIONAL задачи либо выполнены, либо явно помечены как DEFERRED

    Модуль считается ЗАКРЫТЫМ только после:
    смысл → границы → код → проверка соответствия.

    Переход к следующему модулю ЗАПРЕЩЁН,
    если текущий модуль существует только «на бумаге»
    или имеет незакрытые обязательные пункты.


────────────────────────────────────────
МОДУЛЬНАЯ СТРУКТУРА (ОБЯЗАТЕЛЬНО)
────────────────────────────────────────

— Все модули хранятся в каталоге /01-modules
— Каждый модуль:
  • имеет УНИКАЛЬНЫЙ порядковый номер
  • номер = логический приоритет разработки
  • имя отражает суть модуля

— В КАЖДОМ модуле ОБЯЗАТЕЛЬНО:

  1. MODULE-SPEC.md
     • философия и смысл
     • этическая роль
     • эмоциональное влияние
     • связи
     • границы и запреты

     MODULE-SPEC.md фиксирует ЗАЧЕМ существует модуль
     и какие действия в нём ЗАПРЕЩЕНЫ навсегда.

     Делай единым текстом в формате .md,
     чтобы можно было скопировать и вставить.

  2. DEVELOPMENT-CHECKLIST.md
     • фазовая логика
     • архитектурные и security checkpoints
     • этические guardrails
     • ограничения для Claude Opus и AI

     DEVELOPMENT-CHECKLIST.md определяет,
     ИМЕЕМ ЛИ МЫ ПРАВО реализовывать и расширять модуль.

     Делай единым текстом в формате .md,
     чтобы можно было скопировать и вставить.

  3. IMPLEMENTATION-CHECKLIST.md
     • конкретные инженерные задачи
     • реальные TODO по коду
     • MUST / SHOULD / OPTIONAL / DEFERRED
     • критерии закрытия модуля

     IMPLEMENTATION-CHECKLIST.md отвечает на вопрос:
     «ЧТО КОНКРЕТНО ДЕЛАЕМ РУКАМИ».

     IMPLEMENTATION-CHECKLIST.md:
     — НЕ может противоречить MODULE-SPEC.md
     — НЕ может нарушать DEVELOPMENT-CHECKLIST.md
     — используется для программирования и закрытия модуля

     Модуль НЕ считается реализованным,
     пока не закрыты все MUST DO пункты
     IMPLEMENTATION-CHECKLIST.md.

     Делай единым текстом в формате .md,
     чтобы можно было скопировать и вставить.

Никакого кода без утверждённого смысла.
Никакой эффективности ценой человека.
