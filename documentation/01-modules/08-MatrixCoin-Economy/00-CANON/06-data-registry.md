# 00-CANON/06-data-registry.md
## Data Registry & Semantic Boundaries

**Статус:** CANONICAL / IMMUTABLE / LOCKED  
**Роль:** Реестр допустимых данных в домене MatrixCoin Economy  
**Принцип:** No data — no meaning  
**Audit — источник истины**  
**Человек — финальный арбитр интерпретаций**

---

## 6.0. Назначение документа

Настоящий документ фиксирует **реестр данных**, используемых в домене MatrixCoin Economy (MC / GMC).

Документ:
- **не описывает механику обработки данных**;
- **не описывает бизнес-логику**;
- **не описывает интерфейсы**.

Цель документа:
- зафиксировать **какие данные существуют**;
- определить **их смысловые границы**;
- исключить появление неявных, производных или интерпретативных данных.

---

## 6.1. Принципиальные положения

### 6.1.1. Только зафиксированные данные существуют

Если данные:
- не описаны в этом реестре,
- не имеют чёткого определения,

→ **они не существуют в домене**.

### 6.1.2. Данные ≠ интерпретация

Система:
- хранит данные;
- не делает выводов;
- не присваивает смысл за человека.

Любая интерпретация данных:
- внешняя;
- человеческая;
- не является частью реестра.

### 6.1.3. Один смысл — одно поле

Каждое поле данных:
- имеет один смысл;
- не допускает множественных трактовок;
- не используется для косвенных выводов.

---

## 6.2. Категории данных

### 6.2.1. Идентификационные данные

Данные, позволяющие однозначно определить субъект или объект.

Примеры:
- уникальный идентификатор пользователя;
- идентификатор кошелька;
- идентификатор события.

Ограничения:
- не используются для оценки;
- не агрегируются в показатели.

---

### 6.2.2. Структурные данные

Данные, описывающие положение в системе.

Примеры:
- принадлежность к домену;
- тип сущности (MC / GMC);
- связь сущностей.

Ограничения:
- не интерпретируются как статус человека;
- не используются для сравнения людей.

---

### 6.2.3. Событийные данные

Данные, фиксирующие факт произошедшего события.

Примеры:
- начисление;
- перевод;
- списание;
- фиксация условия.

Свойства:
- append-only;
- неизменяемы;
- имеют временную метку.

---

### 6.2.4. Агрегированные данные

Производные данные, полученные путём арифметической агрегации событий.

Примеры:
- текущий баланс;
- суммарный оборот;
- количество событий.

Ограничения:
- не считаются оценкой;
- не используются для принятия решений автоматически;
- всегда воспроизводимы из событий.

---

### 6.2.5. Метаданные

Данные о данных.

Примеры:
- источник события;
- версия правила;
- контекст фиксации.

Назначение:
- обеспечение audit;
- трассируемость;
- воспроизводимость.

---

## 6.3. Запрещённые типы данных

В домене MatrixCoin Economy **запрещено хранить**:

- психологические оценки;
- мотивационные профили;
- поведенческие рейтинги;
- прогнозы поведения;
- интерпретации намерений;
- «оценки полезности» человека.

---

## 6.4. Связь данных с MC / GMC

### 6.4.1. MC

Допустимые данные:
- факт начисления;
- факт использования;
- остаток;
- история событий.

Недопустимо:
- денежный эквивалент;
- «ценность» MC;
- сравнение между людьми.

---

### 6.4.2. GMC

Допустимые данные:
- факт владения;
- условия получения;
- события движения.

Недопустимо:
- капитализация;
- оценка в деньгах;
- прогноз стоимости.

---

## 6.5. Временные характеристики данных

Все данные:
- имеют временную метку;
- не переписываются;
- не корректируются задним числом.

Запрещено:
- изменять прошлые значения;
- пересчитывать историю под новые правила;
- «уточнять» старые данные.

---

## 6.6. Доступ и видимость данных

Данные:
- разделены по уровням доступа;
- видимы только в допустимом контексте.

Принцип:
- **минимально необходимая видимость**.

AI:
- имеет read-only доступ;
- не видит персональные контексты;
- не агрегирует данные для выводов.

---

## 6.7. Границы использования данных

Запрещено:
- использовать данные для давления;
- использовать данные для манипуляции;
- использовать данные вне заявленного смысла.

Разрешено:
- хранение;
- отображение;
- проверка;
- audit.

---

## 6.8. Эволюция реестра данных

Изменение реестра:
- считается изменением CANON;
- требует отдельной фиксации;
- не может происходить неявно.

Добавление нового поля данных:
- допускается только через обновление CANON;
- без обратной силы.

---

## 6.9. Нарушения и последствия

Нарушением считается:
- появление неописанных данных;
- скрытая агрегация;
- интерпретация на уровне данных.

Последствия:
- перевод домена в статус **DEGRADED** или **INVALID**;
- восстановление только вручную;
- обязательная фиксация причины.

---

## 6.10. Финальная фиксация

Этот документ:
- определяет пределы данных;
- защищает от смыслового дрейфа;
- исключает «умные данные».

> **Если данных нет в реестре —  
> они не имеют права на существование.**
